# Zomboid_Migration
Project Zomboid Migration mod

# Техническая документация проекта "Realistic Forest Migration"
**Разработка модификации системы миграции зомби для Project Zomboid (Build 41) на стыке Java и нативного C++ движка.**

---

## 1. ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ)

### 1.1. Введение и актуальность исследования
В текущей реализации архитектуры игры Project Zomboid (Build 41) подсистема мета-популяции зомби использует математическую модель равномерного перераспределения виртуальных сущностей между макро-ячейками (cells) карты. Данный алгоритм направлен на поддержание постоянного игрового напряжения посредством заполнения зачищенных зон мигрирующими группами из соседних закрытых областей.

Однако при установке экстремальных множителей популяции (режим «Песочницы» с множителем X = 16.0) стандартный алгоритм демонстрирует логическое противоречие: зачищенные игроком труднопроходимые лесные массивы за короткий промежуток времени заполняются тысячами автономных сущностей из смежных секторов. Это полностью нивелирует стратегическую ценность выживания вне городских агломераций, ломает игровую логику реализма и вызывает избыточную нагрузку на процессор из-за непрерывного спавна/деспавна сотен 3D-моделей персонажей.

**Цель работы:** Проектирование и программная реализация мода на языке Lua, модифицирующего логику мета-миграции игровых сущностей для обеспечения возможности долговременного удержания и очистки изолированных и лесных зон без нарушения общего баланса симулятора.

### 1.2. Функциональные требования к программному модулю
* Динамическая фильтрация по лесистости (Forest Density Filter): Модуль должен выполнять потоковый анализ структуры тайлов целевого чанка. Если коэффициент плотности деревьев превышает заданный порог, входящий поток миграции должен снижаться по экспоненциальной зависимости.
* Эффект «Памяти очистки» (Zone Security Buffer): Создание временного триггерного буфера в радиусе активности игрока. Ячейки, в которых зафиксировано массовое уничтожение зомби игроком, должны блокировать входящую миграцию на фиксированный период времени для симуляции реального истощения локального пула.
* Консервация пула (No In-Air Spawning): Запрет скрытой генерации зомби внутри закрытых ячеек, не имеющих физической связи с донорскими городскими зонами. Миграция должна строго интерпретироваться движком как пространственное перемещение существующих виртуальных объектов, а не как отложенный респавн.
* Оптимизация высоконагруженных вычислений: Алгоритм проверки не должен вызывать каскадных задержек (микрофризов) главного потока при итерации по массивам объектов на популяции х16. Применяется событийная модель обработки вместо безусловного ежесекундного тика.

### 1.3. Архитектура и ограничения интеграции
Подсистема популяции игры имеет гибридную архитектуру. Вычислительное ядро скомпилировано в нативные библиотеки (C/C++), а интерфейсный слой реализован на Java. Интеграция Lua-мода будет осуществляться посредством перехвата функций Java-слоя, выступающих мостом (JNI Wrapper) для нативного кода. Прямая модификация бинарных файлов PZPopMan64.dll/.so исключается в целях сохранения кроссплатформенности и совместимости со Steam Workshop.

## 2. АНАЛИЗ ИСХОДНОГО КОДА ЯДРА ПОПУЛЯЦИИ (JAVA)

Для проектирования точек интеграции (хуков) был проведен декомпиляционный анализ ключевых компонентов пакета zombie.popman.

### 2.1. Класс ZombiePopulationManager
Класс представляет собой основной управляющий компонент, координирующий состояние виртуальных и реальных зомби. Анализ исходного байт-кода выявил критическую архитектурную особенность — передачу вычислений в нативный С++ код через интерфейс JNI. Ключевые декларации низкоуровневых функций:

* private static native void n_init(boolean var0, boolean var1, int var2, int var3, int var4, int var5);
* private static native void n_config(float var0, float var1, float var2, int var3, float var4, float var5, float var6, float var7, int var8);
* private static native void n_updateMain(float var0, double var1);
* protected static native void n_spawnHorde(int var0, int var1, int var2, int var3, float var4, float var5, int var6);

Математическая инициализация параметров конфигурации миграции выполняется через метод onConfigReloaded(), который транслирует параметры Sandbox-настроек в нативный метод n_config:

* Код метода: if (!this.bClient) { SandboxOptions.ZombieConfig var1 = SandboxOptions.instance.zombieConfig; n_config(var1.Multiplier, var1.StartMultiplier, var1.PeakMultiplier, var1.PeakDay, var1.RespawnHours, var1.RespawnUnseenHours, var1.RespawnMultiplier, var1.RedistributeHours, var1.FollowSoundDistance); }

* Критическая точка интеграции №1: Управление параметром RedistributeHours (интервал миграции в часах). Изменяя динамически данную переменную через Lua-окружение для конкретных зон, мы можем принудительно останавливать или замедлять циклы миграции.

Перевод зомби из виртуального состояния (абстрактные цифры в нативной памяти) в физические объекты сцены осуществляется методами:
* public void addZombieStanding(float var1, float var2, float var3, int var4)
* public void addZombieMoving(float var1, float var2, float var3, int var4, IsoDirections var5)

* Критическая точка интеграции №2: Перехват данных методов позволяет выполнить валидацию координат (X = var1, Y = var2) непосредственно перед конструированием объекта IsoZombie.

### 2.2. Класс ZombieCountOptimiser
Отвечает за принудительное уничтожение (куллинг) удаленных или избыточных сущностей в целях оптимизации потоков рендеринга.
* Логика метода incrementZombie: если zombieCountForDelete > 0 и выполняется проверка canBeDeletedUnnoticed(20.0F), то зомби добавляется в список на удаление zombiesForDelete.

* Анализ применения: Метод использует предикат var0.canBeDeletedUnnoticed(20.0F) (возможность удаления без визуального обнаружения игроком на расстоянии от 20 тайлов). Мод может использовать эту логику для принудительной деактивации зомби, незаконно мигрировавших в лесную буферную зону.

### 2.3. Класс ZombiePopulationRenderer
Хотя данный класс предназначен для визуализации отладочной информации, он раскрывает внутренние константы адресации пространства:
* protected static final int SQUARES_PER_CELL = 300;

Это фиксирует математическую сетку: 1 ячейка (cell) равна области в 300 х 300 тайлов. Метод renderCell вызывает нативные функции сбора метрик:
* int var23 = this.n_getRealZombieCount(var3, var4);
* int var24 = this.n_getVirtualZombieCount(var3, var4);

* Анализ применения: Эти вызовы подтверждают доступность методов получения точного объема виртуальной популяции из Java-контекста, что дает возможность Lua-скрипту запрашивать данные в конкретных координатах для аналитики эффективности фильтра.

## 4. МАТЕМАТИЧЕСКАЯ МОДЕЛЬ ФИЛЬТРА

Допустим, стандартный шаг миграции пытается перевести объем зомби V_init в выбранную ячейку. Мы вводим коэффициент лесистости ячейки K_f, который рассчитывается как:

K_f = N_tree / N_total

Где N_tree — количество тайлов деревьев и густой растительности в чанке, а N_total — общее количество тайлов в чанке (10 х 10 = 100). Итоговый объем спавна мигрирующих зомби в лесу V_final будет определяться уравнением:

V_final = V_init * ((1 - K_f) ^ alpha)

Где alpha — коэффициент жесткости фильтрации (рекомендуемое значение для популяции х16: alpha = 3.0). Таким образом, при плотности леса в 80% (K_f = 0.8), входящий поток зомби будет снижен более чем на 99%, что обеспечит полную изоляцию лесного лагеря игрока.
